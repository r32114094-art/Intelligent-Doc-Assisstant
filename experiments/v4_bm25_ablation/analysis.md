## 核心分析：增量贡献分解

### 1. BM25 的增量贡献

| 对比 | CR 变化 | 综合变化 |
|---|---|---|
| Dense Only → Hybrid (Dense+BM25) | 3.70 → **4.10** (+0.40) | 4.45 → 4.47 (+0.02) |
| Dense+MQE+HyDE → Hybrid+MQE+HyDE | 4.05 → 3.90 (-0.15) | 4.52 → 4.50 (-0.02) |

**结论**：BM25 在基线场景（无 MQE/HyDE）提供了 **+0.40 CR 的增量**（+10.8%），但综合分仅微增 +0.02（AR 下降抵消了 CR 提升）。
当叠加 MQE+HyDE 后，BM25 的贡献反而变为**轻微负向**（CR -0.15），说明 MQE 生成的扩展查询已覆盖了 BM25 的关键词召回能力，两者存在功能冗余。

### 2. MQE+HyDE 的增量贡献

| 对比 | CR 变化 | 综合变化 |
|---|---|---|
| Dense Only → Dense+MQE+HyDE | 3.70 → **4.05** (+0.35) | 4.45 → **4.52** (+0.07) |
| Hybrid → Hybrid+MQE+HyDE | 4.10 → 3.90 (-0.20) | 4.47 → 4.50 (+0.03) |

**结论**：MQE+HyDE 在纯 Dense 场景提供 **+0.35 CR 增量**，综合分提升 +0.07。叠加 BM25 后 CR 反而下降 -0.20，进一步验证了两者的功能冗余。

### 3. 叠加效果：是否超线性？

```
BM25 单独增量 (CR): +0.40
MQE+HyDE 单独增量 (CR): +0.35
两者之和: +0.75
实际叠加增量 (Hybrid+MQE+HyDE vs Dense Only): +0.20

+0.20 << +0.75 → 严重次线性，两者功能高度重叠
```

**结论**：BM25 和 MQE+HyDE **不是互补关系，而是替代关系**。

## 核心发现

### ✅ 发现 1: BM25 在简单检索中有效

BM25 在无高级检索（MQE/HyDE）时，对 CR 有明显提升：
- easy: 3.40 → 4.00 (+0.60)
- medium: 4.10 → **4.60** (+0.50)
- hard: 3.20 → 3.20 (无变化)

BM25 对包含明确关键词的问题（easy/medium）效果好，对需要跨 chunk 推理的困难题无帮助。

### ✅ 发现 2: FF 全面稳定在 5.0 附近

四组配置的 FF 均为 4.95~5.00，验证了 V3 评估管道修复的稳定性——LLM 完全忠实于检索上下文。

### ⚠️ 发现 3: 综合最佳配置是 Dense+MQE+HyDE

| 配置 | 综合 | 成本效益 |
|---|---|---|
| Dense Only | 4.45 | ⭐⭐⭐⭐⭐ 最低延迟 17.1s |
| **Dense+MQE+HyDE** | **4.52** | ⭐⭐⭐ 延迟 35.5s |
| Hybrid (Dense+BM25) | 4.47 | ⭐⭐⭐⭐ 延迟 20.1s |
| Hybrid+MQE+HyDE | 4.50 | ⭐⭐ 最高延迟 38.6s |

- 如果追求**最高质量**：Dense+MQE+HyDE (4.52)
- 如果追求**性价比**：Hybrid (Dense+BM25) (4.47, 延迟仅 20s)
- 如果追求**最低延迟**：Dense Only (4.45, 延迟 17s)

### 📊 发现 4: 困难题 CR 仍是天花板

所有四组配置在 hard 难度的 CR 均为 3.20，**完全相同**。说明困难题瓶颈不在检索策略，而在：
- chunk_size 过大导致相关信息分散
- 文档本身对跨章节问题的信息密度不足

## 总结与建议

### 1. BM25 价值定位
- ✅ 在**低成本**场景（不启用 MQE/HyDE）下，BM25 是性价比最高的增强手段（+0.40 CR，仅 +3s 延迟）
- ⚠️ 与 MQE+HyDE 叠加时无增量甚至轻微有害
- 建议：**保留 BM25 作为默认开启的基础增强**，仅在启用 MQE+HyDE 时可考虑关闭以节省资源

### 2. 推荐生产配置
- **标准模式**：Hybrid (Dense+BM25)，综合 4.47，延迟 20s
- **高质量模式**：Dense+MQE+HyDE，综合 4.52，延迟 35s

### 3. V5 方向
- **P0**: 困难题 CR 优化 — 考虑减小 chunk_size（如 400~600）或引入 Parent-Child 分块策略
- **P1**: 替换中文 Reranker（如 `BAAI/bge-reranker-base`），重新评估 Rerank 的价值
- **P2**: 引入独立 Judge 模型（如 GPT-4o-mini）消除自评偏见
